Overview

This unified compiler processes a custom programming language with strong type checking and supports both interpretation and semantic analysis. The system demonstrates key compiler construction concepts from tokenization through execution.

Built as an educational project to understand the internals of programming language implementation, this compiler showcases modern compiler design patterns and best practices in software engineering.

Compiler Architecture

Lexical Analyzer (scanner.l)

Tokenizes source code, recognizing keywords, operators, literals, and identifiers

‚Üì

Syntax Analyzer (parser.y)

Builds parse tree using context-free grammar with proper operator precedence

‚Üì

Semantic Analyzer (types.cc)

Validates types, checks for narrowing, ensures variable declarations, and type consistency

‚Üì

Interpreter (types.cc)

Directly executes type-checked program and produces results

The compiler follows a traditional four-phase architecture, with each phase building upon the work of the previous one. Flex and Bison handle the lexical and syntactic analysis, while custom C++ code implements type checking and interpretation.

Compilation Modes

Full Mode (Default)

Complete compilation with type checking and interpretation

./compile < program.txt

Type-Check Only

Validates syntax and semantics without execution

./compile --type-check < program.txt

Interpret Only

Skips type checking for faster execution of known-valid programs

./compile --interpret < program.txt

Comprehensive Error Detection

The compiler performs thorough validation at every phase, catching errors early and providing clear feedback:

  • Malformed literals

Syntax Errors

  • Missing semicolons
  • Unbalanced parentheses
  • Grammar violations

Semantic Errors

  • Type narrowing detection
  • Undeclared variables
  • Duplicate identifiers
  • Type mismatches

Status & Future Enhancements

‚úì Stable The compiler is fully functional with comprehensive features for educational and experimental use.

Known Limitations

Switch Statement Fallthrough

Cases currently fall through to 'others' clause. Workaround: use if/elsif chains

No Function Calls

Functions can be defined but not called within the language. Single-function programs only

No I/O Operations

Programs use command-line parameters for input and print final result only

Planned Enhancements

  • Fix switch statement case matching behavior
